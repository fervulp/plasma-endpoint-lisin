# LiSin — карта системы

Документ для того, кто видит проект впервые (человека или другой чат Claude).
Отвечает на «что здесь есть, зачем каждая часть и куда смотреть».
Составлен **измерением работающей системы**, а не по памяти; цифры — на
2026-07-20 и будут дрейфовать, структура — нет.

Хронология изменений — в `CLAUDE.md`. Правила, которые нельзя нарушать —
в `expertise/PRINCIPLES.md`, **сверяться с ними до любой правки**.
Контракт для автора экспертизы — `expertise/HOWTO.md`.

---

## 1. Что это

Лёгкий локальный EDR для Fedora на одном ноутбуке. Python 3 (stdlib) +
PySide6/Qt Quick + Kirigami. Без сервера, **без root**, без внешних сервисов
в основном контуре.

Отвечает на три вопроса:

| Вопрос | Где | Хранилище |
|---|---|---|
| Что в системе есть **сейчас** | «Состояние» | `state.db` (снимок, upsert) |
| Что **произошло** | «События» | `events.db` (поток, append-only) |
| Что с этим **делать** | «Дашборды» → находки, граф | считается на лету |

Запуск (обязательно `PYTHONNOUSERSITE=1`, иначе pip-PySide6 не видит
системный Kirigami):

```
PYTHONNOUSERSITE=1 QT_QUICK_CONTROLS_STYLE=org.kde.desktop python3 lisin_app.py
```

Один экземпляр: `flock` на `~/.local/share/lisin/lisin.lock`.
**Грабля, стоившая много времени:** запущенный старый экземпляр каждые 2 с
переписывает БД своим кодом. При «не работает» первым делом сверяйте время
старта процесса с временем правок.

## 2. Главная идея: всё идёт через конвейер

```
точка входа  →  нормализация  →  обогащение  →  фильтр  →  точка выхода
 (bash-cmd)    (Python-плагин)   (Python-плагин)          (statedb|events)
```

Каждый узел — **объект экспертизы**, YAML-файл с кодом внутри. Приложение —
это движок; вся предметная логика лежит в `expertise/` и правится без
открытия исходников. Это положение 1–2 и причина, по которой новый источник
добавляется без единой строки в Python.

- нормализация обязана определить `normalize(text) -> list[dict]`;
- обогащение — `enrich(rows) -> rows`;
- колонки таблицы выводятся из ключей словаря, а не задаются;
- таблица/ключ задаются в **точке выхода**, а не в нормализации.

**Доверие:** код правила исполняется, как и bash точек входа. Импорт чужой
экспертизы = запуск чужого кода.

### Разделение движка и данных
`findings.py`, `correlate.py`, `pipeline.py` — **движки**: они ничего не знают
о конкретных пакетах, портах и путях. Все конкретности — в `expertise/`.
Нарушение этого и есть главный дефект, который стоит искать при ревью.

## 4. API для QML — `agent/api/`

`Backend(QObject, StateApi, EventsApi, DashboardApi, ExpertiseApi, SystemApi)`.
Миксины — обычные классы; `@Slot` в них регистрируется при наследовании.
`lisin_app.py` содержит только планировщик, сбор ошибок и `main()`.

| Миксин | Слотов | Про что |
|---|---:|---|
| `expertise.py` | 28 | конвейеры (граф, черновик, запуск, peek), каталог экспертизы, CRUD, прогон правил |
| `dashboard.py` | 23 | дашборд, разбор процесса, **граф и пивот**, находки, уязвимости, панели, WHOIS |
| `events.py` | 16 | лента, статистика, произвольный SELECT, группировки, история SQL, сохранённые запросы, цепочки |
| `state.py` | 10 | снимок состояния, правки таблиц/ячеек, SQL read-only |
| `system.py` | 6 | ошибки, метрики, настройки |

**Ключевые слоты графа** (самое новое и самое хрупкое):
`processLinks(pid, expanded)`, `anchorGraph(kind, val, expanded)`,
`anchorList(kind)`, `nodeInfo(node)`.

> **Грабля, уже стоившая поломки дашборда:** JS-массив приходит из QML как
> `QJSValue`, он **не итерируется**. Разворачивать `.toVariant()`
> (`DashboardApi._str_list`). Слот, который зовёт QML, **проверять через QML**,
> а не вызовом из Python со списком — из Python баг не воспроизводится.

### Как устроен граф расследования (ядро UX)
- **Якорь** — сущность любого типа: процесс, приложение, порт, пользователь,
  конфиг, открытый файл. Один движок `anchor_graph`.
- **Лесенка**: якорь сверху, под ним блоки-категории сверху вниз —
  приложение → служба → сеть → файлы → конфигурация → …
- **Блок сворачивается** в мета-узел со счётчиком (>4 элементов). Раскрытие
  показывает **все** элементы; рёбра идут **от блока**.
- Слова-типы («процесс», «событие») убраны — тип читается по иконке блока.
- Кнопки на узле: сделать центром (пивот), открыть в «Состоянии», события,
  WHOIS. Есть полноэкранный режим и хлебные крошки пивота.
- **Раскладка (x/y) считается в Python**, QML только рисует — иначе узлы
  прыгают между перерисовками.

## 8. Как проверять (обязательно перед «готово»)

Компиляция QML и offscreen-рендер **не ловят** ошибки слотов и биндингов.
Минимальный набор:

1. `python3 -m py_compile lisin_app.py agent/*.py agent/api/*.py`
2. компиляция **всех** `ui/*.qml` разом (`QQmlComponent` в цикле);
3. **прогон через QML**: загрузить вид в `QQuickView`
   (`setResizeMode(SizeRootObjectToView)` — иначе окно подгоняется под
   корень и геометрия врёт), вызвать его функции, собрать предупреждения
   через `qInstallMessageHandler`;
4. прогон конвейера — ошибок узлов быть не должно;
5. `bash packaging/build-rpm.sh`.

Харнессы живут в scratchpad сессии (`qml_all.py`, `dash_flow.py`,
`ladder.py`, `fullscreen.py`) — их дешевле пересоздать, чем искать.

## 9. Известные ограничения и долги — честно

**Ограничения по природе (не баги):**
- Без root: `/proc/<pid>/exe` чужого процесса не читается; правил аудита ядра
  нет; объём трафика в байтах недоступен — считаем **сессии**, и это написано
  в интерфейсе.
- `open_files` — **снимок открытого**, а не журнал обращений. Процесс, который
  читает файл и сразу закрывает, между снимками не проявится.
- Поллинг: короткоживущие процессы и соединения короче интервала пропускаются.
- Связь «сессия → процесс» идёт по `tty`/`user`, помечается как вероятная.

**Долги (нарушают положения, не закрыты):**
- **`c2_contact` — мёртвый детект.** Условие `threat_indicator <> ''`, а
  обогащение `ip_threat` удалено вместе с фидами. Совпадений 0 и не будет.
  Даёт ложное чувство «мы ловим C2» — либо вернуть источник, либо убрать.
- **`chains.py`** — правила связывания живут в коде, а не в экспертизе
  (положение 2). Механизм рабочий, из UI скрыт.
- **`ipintel.py`** ходит в сеть из плагина обогащения, минуя конвейер
  (положение 1): точку входа не видно, нельзя выключить и прогнать вручную.
- `proc_cmd_risk` — маркеры-регулярки в обогащении, им место в `detections/`.
- Таблица `containers` пуста (docker/podman не запущены) — источник не
  проверен на живых данных.
- ~20 таблиц вида `item/value` (`system`, `cpu`, `memory`, `boot`,
  `security`, `swap_info`) **намеренно** не связаны: это свойства машины
  целиком, у них нет сущности для связи. Притягивать их было бы враньём.

**Повторяющиеся грабли (все уже случались):**
- `%{NAME}` внутри итерации rpm по массиву не разворачивается — нужен `%{=NAME}`.
- YAML: regex со скобками в двойных кавычках ломает парсер — только одинарные.
- QML: `Kirigami.Dialog` — это Popup, открывается `open()`, а не биндингом
  `visible`; второй `Component.onCompleted` в файле = ошибка; `availableWidth`
  брать по id самого `ScrollView`.
- Одна величина в двух местах должна считаться **одним** запросом: разные
  `LIMIT` уже давали «в графе 110 файлов, в панели 60».
- Любой `LIMIT`, который может сработать, обязан быть виден в интерфейсе.
